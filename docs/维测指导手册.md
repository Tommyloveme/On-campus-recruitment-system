# 校招入职跟踪管理系统 — 维测指导手册

> 本系统面向**已接收 Offer** 的校招候选人，跟踪签约、体检、入职预约到正式入职的全流程。

> 面向系统维护人员与测试人员，涵盖架构、配置、接口、数据库、部署、排障与测试方法。
> 用户操作说明见网页右上角「操作手册」（`static/manual.pdf`，由 `tools/gen_manual.py` 生成）。

## 1. 系统架构

```
浏览器（原生 HTML/CSS/JS，无构建依赖）
   │  HTTP/JSON (Session Cookie 认证)
   ▼
waitress（多线程 WSGI，默认32线程） ── Flask 应用（app.py，单文件后端）
   │
   ├── SQLite（data/candidates.db，WAL 模式）
   ├── 简历文件（data/resumes/<候选人id>_<随机>.pdf|.docx）
   └── 配置文件（config/*.json）
```

- **技术栈**：Python 3.9+ / Flask / SQLite（标准库 sqlite3）/ openpyxl（Excel）/ mammoth（docx 转 HTML）/ reportlab（生成手册 PDF）/ waitress（生产 WSGI）。前端 Chart.js 已离线内置（`static/vendor/`）。
- **并发模型**：waitress 多线程 + 每请求独立 SQLite 连接 + WAL 日志模式（读写不互斥）+ busy_timeout 写锁等待，支撑 500+ 并发用户。

## 3. 配置说明

### 3.1 app_config.json（重启生效；ui 部分刷新页面即生效）

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `server.port` | 8000 | 监听端口（环境变量 `PORT` 优先） |
| `server.threads` | 32 | waitress 工作线程数（环境变量 `THREADS` 优先） |
| `server.max_upload_mb` | 16 | 上传文件大小上限 |
| `ui.page_size` | 15 | 候选人列表默认每页条数 |
| `ui.page_size_options` | [15,30,50,100,0] | 每页条数可选项（0=全部） |
| `ui.clip_max_width` | 320 | 表格长内容截断宽度（px），悬浮仍可见全文 |
| `backup.interval_hours` | 1 | 自动备份间隔（小时） |
| `backup.retention_days` | 3 | 备份保留天数（过期自动清理） |
| `security.default_password` | 123456 | 新建用户的默认密码 |
| `resume.allowed_ext` | [.pdf, .docx] | 允许上传的简历格式 |
| `logs.page_size` | 30 | 操作日志每页条数 |

### 3.2 fields.json（刷新页面即生效）

每个字段支持：`key`（内部名，不可随意改，改了旧数据对应不上）、`label`（显示名）、`excel_column`（导入表头）、`type`（text/date/select）、`options`（下拉选项）、`colors`（取值→徽章颜色 gray/blue/green/yellow/red）、`visible`/`editable`/`importable`/`required`/`multiline`。

- **新增字段**：在 `fields` 数组中追加一项即可，导入模板、表格列、编辑表单、图表维度全部自动生效。
- **删除字段**：从数组中移除；历史数据中该 key 仍保留在 JSON 中但不再展示/导入。
- **分组覆盖**：`fields_group_<id>.json` 仅含 `visible` 映射，优先级高于全局配置；删除该文件即恢复全局默认。

## 4. 数据库

SQLite，4 张表（建表语句见 `app.py` 中 `SCHEMA`）：

| 表 | 关键字段 | 说明 |
| --- | --- | --- |
| `groups` | id, name | 权限分组 |
| `users` | username, password_hash, role(admin/group_admin/editor/viewer), group_id | 用户与角色 |
| `candidates` | group_id, data(JSON), resume_file, resume_name | 候选人；业务字段全部存于 data JSON，结构由 fields.json 驱动 |
| `logs` | user_name, action, candidate_id, group_id, message | 操作日志（create/update/delete/import/export/user/group/config） |

- 升级兼容：启动时 `_migrate()` 自动为老库补列（如 resume_file）。新增列时在该函数中追加 `ALTER TABLE`。
- **自动备份**：服务启动后立即备份一次，之后每小时（`backup.interval_hours`）用 SQLite 在线备份 API 生成快照到 `data/backups/backup_<时间戳>.db`，保留 3 天（`backup.retention_days`）后自动清理。管理员可在「系统管理 → 数据备份与恢复」中手动备份或一键恢复任意快照；恢复前会自动保存当前状态。注意：备份只覆盖数据库，不含 `data/resumes/` 简历文件。
- 整机备份：直接复制 `data/` 目录（建议停服或低峰期执行；WAL 模式下需同时复制 `.db-wal`/`.db-shm` 或先执行 `PRAGMA wal_checkpoint`）。

## 5. 接口清单

认证方式：Session Cookie；未登录返回 401，无权限返回 403，业务错误返回 400+`{"error": "..."}`。

| 方法 | 路径 | 权限 | 说明 |
| --- | --- | --- | --- |
| POST | /api/login | 公开 | 登录 |
| POST | /api/logout | 登录 | 退出 |
| GET | /api/me | 登录 | 当前用户信息 |
| GET/PUT | /api/config | 登录 / admin+组管理员 | 字段配置读取/保存（?group_id= 区分范围） |
| GET/POST | /api/groups | 登录 / admin | 分组列表 / 创建 |
| DELETE | /api/groups/{id} | admin | 删除空分组 |
| GET/POST | /api/users | admin、组管理员 | 用户列表 / 创建（组管理员限本组、限组成员/只读） |
| PUT/DELETE | /api/users/{id} | admin | 修改 / 删除用户 |
| GET/POST | /api/candidates | 登录 | 列表（自动按组过滤）/ 新增 |
| PUT/DELETE | /api/candidates/{id} | 编辑权 / 删除权 | 修改（自动记录字段级变更日志）/ 删除 |

| GET | /api/import/template | 登录 | 下载导入模板（按字段配置生成） |

## 6. 权限模型

| 操作 | admin | group_admin | editor | viewer |
| --- | :-: | :-: | :-: | :-: |
| 查看候选人 | 全部 | 本组 | 本组 | 本组 |
| 新增/编辑 | 全部 | 本组 | 本组 | × |
| 删除候选人/简历 | 全部 | 本组 | × | × |
| 数据图表 | 全部 | 本组 | × | × |
| 添加用户 | 任意角色 | 本组成员/只读 | × | × |
| 字段配置 | 全局+各组 | 本组 | × | × |

核心校验函数：`can_view_group / can_edit_group / can_delete_group`（`app.py`），前端按钮显隐只是体验优化，**安全以后端校验为准**。

## 7. 部署与运维

```bash
# Windows
start.bat            # 或 start.bat --demo 写入演示数据
# SUSE Linux
chmod +x start.sh && ./start.sh
# 修改端口/线程（任选其一）
PORT=9000 THREADS=64 ./start.sh        # 环境变量
# 或编辑 config/app_config.json 后重启
```

- 服务日志直接输出到控制台；Linux 建议用 systemd/nohup 托管并重定向输出。
- 升级步骤：备份 `data/` → 替换代码 → `pip install -r requirements.txt` → 重启（数据库自动迁移）。
- 重置管理员密码：删除该用户行后重启会重建默认 admin（仅当 users 表为空），或直接用 Python 调 `generate_password_hash` 更新该行。

## 8. 测试指导

### 8.1 冒烟测试（推荐每次改动后执行）

```bash
python app.py --demo     # 终端1：启动带演示数据的服务
python smoke_test.py     # 终端2：运行全量冒烟测试
```

覆盖范围：登录鉴权、字段配置数量、候选人 CRUD 与字段级变更日志、Excel 导入（含 120 人批量）、
简历上传/预览/更换/下载/删除/打包导出、四级角色权限边界（组成员无删除权、组管理员加人/配置本组字段、
默认密码登录）、按组字段配置生效、数据备份/恢复（含非法备份名拦截）、并发 60 会话压测。**全部 PASS 即可发布。**

### 8.2 手工回归要点

1. 列筛选多列组合 + 全局搜索叠加；日期列排序；列宽拖拽。
2. 分页：默认 15 条/页，切换每页条数、翻页、筛选后回到第一页。
3. 「当前进展」更新按钮自动追加今日前缀（如 0612：）。
4. 勾选导出 Excel / 简历 zip；下载导入模板再导入回去（应全部匹配为更新）。
5. 数据图表：日期维度三种粒度（年月日/年月/月份）切换；组管理员只见本组。
6. 右上角操作手册 PDF 可打开。

### 8.3 性能验证

`smoke_test.py` 第 10 节为 60 并发会话（登录+查询）压测，正常应在数秒内全部成功。
更大规模可用 `ab`/`locust` 对 `GET /api/candidates` 压测，瓶颈通常在密码哈希（仅登录时）与磁盘 IO。

## 9. 常见故障排查

| 现象 | 排查方向 |
| --- | --- |
| 端口被占用 / 启动即退 | `netstat -ano \| findstr :8000`（Linux：`ss -lntp`），杀掉残留 python 进程再启动 |
| 请求 404 但代码里有路由 | 多个旧服务实例同时监听端口，请求被旧进程接管；清理所有监听进程后重启唯一实例 |
| database is locked | 确认 WAL 已启用（启动时设置）；检查是否有外部工具长期占用写事务 |
| docx 预览 500 | 文件并非真正的 docx（如改后缀的 doc）；提示用户下载原文件查看 |
| 中文 PDF 手册乱码 | 重新运行 `tools/gen_manual.py`；确认 reportlab 已安装 |
| 修改 fields.json 后页面异常 | JSON 语法错误（用编辑器校验）；`key` 重复或被改名导致旧数据对应不上 |
| 会话全部失效 | `data/.secret_key` 被删除或变更（重启后会重新生成，属正常现象） |

## 10. 代码导读（二次开发）

- `app.py` 按区块组织：基础设施 → 数据库初始化 → 权限 → 日志 → 认证 → 字段配置 → 分组 → 用户 → 候选人 → 简历 → 导入/导出 → 日志查询 → 总览。每个路由函数的 docstring 说明用途与权限。
- `static/app.js` 按页面组织：基础工具（api/toast/modal）→ 登录 → 主框架（tab 路由）→ 候选人页（筛选/排序/分页/列宽/选择）→ 简历操作 → 导入导出 → 数据图表 → 日志 → 系统管理。
- 新增一个业务字段：改 `config/fields.json` 即可；新增一种角色：后端改 `VALID_ROLES` 与三个 `can_*` 函数，前端改 `ROLE_NAMES` 与按钮显隐逻辑。
